## 1. LLM Wiki でできること

LLM Wiki (エルエルエム・ウィキ) は、自分の手元にある資料 (PDF、Word、Markdown、Web 記事など) を AI に読み込ませると、自動で **構造化された個人 Wiki** を作って育ててくれるデスクトップアプリです。

### 1.1 「LLM が wiki を書く / 人間が源泉 (ソース) を集める」

役割分担はシンプルです。

- **あなた (人間) の仕事**: 読みたい資料を集めて投入する。気になることを質問する。
- **LLM (大規模言語モデル、英: Large Language Model) の仕事**: 資料を読み、要約・相互参照・矛盾の指摘・索引づくりといった面倒な作業をすべて引き受ける。

LLM Wiki が自動で書いてくれる Wiki ページは、Markdown 形式のテキストファイルとしてあなたの PC に保存されます。中身を直接編集することも可能ですが、基本的には触らなくても勝手に育っていきます。

### 1.2 RAG との違い — 累積する知識ベース

ChatGPT のファイルアップロードや NotebookLM のような **RAG** (英: Retrieval-Augmented Generation、必要な部分を毎回検索して回答する仕組み) は便利ですが、毎回ゼロから資料を読み直すため、知識が積み上がりません。

LLM Wiki は違います。資料を投入するたびに **「Wiki」という持続的なまとめ** が手元で育っていきます。新しい資料を入れると、関連する既存ページが更新され、矛盾が指摘され、相互リンクが張り直されます。質問は Wiki に対して投げるので、回答はあなたが今までに集めた知識の総体に基づきます。

### 1.3 5 つのテンプレート

LLM Wiki には 5 種類のシナリオテンプレートが付属しています。プロジェクト作成時に 1 つ選ぶだけで、用途に合わせた構造 (ディレクトリ・ページタイプ・命名規則) が自動セットアップされます。

| テンプレート | アイコン | こんな人に |
|---|---|---|
| リサーチ | 🔬 | 論文・記事を読み込んで仮説を更新したい研究者・学習者 |
| 読書 | 📚 | 一冊の本を多角的にメモしたい読書家 |
| 自己成長 | 🌱 | 目標・習慣・振り返りをセルフコーチング的に管理したい人 |
| ビジネス | 💼 | ミーティング・意思決定・プロジェクト文脈を蓄積したいチーム |
| 汎用 | 📄 | 用途未確定 / とりあえず触ってみたい人 |

> 💡 迷ったら **汎用** で始めて、後から `purpose.md` (Wiki の目的を書くファイル) と `schema.md` (Wiki の構造を書くファイル) を編集して用途特化させても構いません。

## 2. 必要な準備

### 2.1 動作環境

LLM Wiki は **macOS / Windows / Linux** に対応したクロスプラットフォームのデスクトップアプリです。

- **macOS**: Apple Silicon (M シリーズ) または Intel (`.dmg` インストーラー)
- **Windows**: 64bit 版 (`.msi` インストーラー)
- **Linux**: `.deb` (Debian/Ubuntu 系) または `.AppImage` (汎用)

ビルド済みバイナリは [GitHub Releases](https://github.com/nashsu/llm_wiki/releases) からダウンロードできます。

### 2.2 LLM プロバイダの選び方

LLM Wiki が AI を使うには、外部の LLM サービス (プロバイダ) と接続する必要があります。以下 4 つの選択肢があります。

| 選択肢 | 費用 | 必要なもの | 特徴 |
|---|---|---|---|
| **Ollama** (ローカル無料) | 0 円 | PC のメモリ 16GB 以上推奨 | 完全ローカルで動作。プライバシー重視。性能はモデル次第。 |
| **Claude Code CLI** | 既存の定額契約があれば追加費用なし | Claude のサブスクリプション + `claude` バイナリ | API キー不要。定額契約が既にあれば最強コスパ。 |
| **OpenAI / Anthropic / Google** (有料 API) | 従量課金 | API キー | 高品質。月数百円〜数千円 (使用量による)。 |
| **OpenAI 互換 API** | 環境による | LM Studio / vLLM / LiteLLM などのエンドポイント URL | 自前サーバーや社内 LLM ゲートウェイを使いたい上級者向け。 |

> 💡 **推奨スターター構成**:
> - お金をかけたくない: macOS + Claude Code CLI (Claude の定額契約をすでに持っているなら) または Ollama
> - 品質重視で気軽に: OpenAI API キーを取得して GPT-4 系を指定
> - プライバシー最優先: Ollama 一択

### 2.3 (任意) ソースから動かしたい場合

開発版を試したい方は Node.js 20 以上と Rust 1.70 以上をインストールしたうえで、次のコマンドで起動できます。

```bash
git clone https://github.com/nashsu/llm_wiki.git
cd llm_wiki
npm install
npm run tauri dev
```

> ⚠️ 通常はビルド済みバイナリを使えば十分です。コマンドラインに不慣れな方は無理にソースビルドする必要はありません。

## 3. 初回起動

### 3.1 アプリを起動する

ダウンロードした `.dmg` / `.msi` / `.deb` / `.AppImage` をインストールし、アプリを起動してください。

### 3.2 自動言語検出

v0.4.0 以降、初回起動時に **OS のロケール** を検出し、日本語環境であれば自動で日本語 UI が選ばれます。後から変更したい場合は **設定 → インターフェース → UI 言語** で切り替えられます。

> 💡 UI の言語と「AI が出力する文章の言語」は別物です。AI 出力の言語は **設定 → 出力設定 → AI 出力言語** で指定します。Auto にしておくと、ソースや質問の言語に追従します。

### 3.3 ウェルカム画面

最初の画面では次の 3 つを選べます。

1. **新規プロジェクト** — 新しい Wiki を作る
2. **プロジェクトを開く** — 既存の Wiki ディレクトリを開く
3. **最近のプロジェクト** — 過去に開いた Wiki の一覧

初回は **新規プロジェクト** を選びますが、その前に LLM プロバイダを設定しておくとスムーズなので、先に [4 章](#4-llm-プロバイダの設定) を進めましょう。

## 4. LLM プロバイダの設定

サイドバーの **設定** (歯車アイコン) → **LLM モデル** カテゴリを開きます。

> 💡 各プロバイダごとに「1 行 = 1 設定」で表示されます。1 つのプロバイダを ON にすると、他は自動的に OFF になります。API キーはプロバイダごとに独立して保存されるため、切り替えても入力した値は失われません。

以下、4 つの代表的な選択肢ごとに設定方法を説明します。

### 4.1 OpenAI を使う

1. [OpenAI のダッシュボード](https://platform.openai.com/api-keys) にログインする
2. 「API Keys」ページで新しい鍵を発行 (`sk-` で始まる長い文字列)
3. LLM Wiki の **設定 → LLM モデル → OpenAI** の行を ON
4. API キー入力欄に貼り付け
5. モデル選択欄から `gpt-4o` などを選ぶ (またはカスタムモデル名を直接入力)

> ⚠️ API キーは他人に漏らさないでください。漏洩した場合は OpenAI 側で速やかに失効させてください。

### 4.2 Anthropic (Claude API) を使う

1. [Anthropic Console](https://console.anthropic.com/) にログイン
2. 「API Keys」から鍵を発行
3. LLM Wiki の **設定 → LLM モデル → Anthropic** の行を ON
4. API キーを貼り付け
5. モデルに `claude-3-5-sonnet-...` などを指定

### 4.3 Ollama (ローカル無料) を使う

1. [Ollama 公式サイト](https://ollama.com/) からインストーラーをダウンロードして起動
2. ターミナルで使いたいモデルを取得 (例: `ollama pull llama3`)
3. LLM Wiki の **設定 → LLM モデル → Ollama** の行を ON
4. URL は通常 `http://localhost:11434` のまま
5. モデル名に `llama3` のように入力

> 💡 7B (70 億パラメータ) クラスのモデルは 16GB 以上のメモリ、13B クラスは 32GB 以上を目安に。動作が遅すぎる場合はもっと小さなモデル (例: `phi3:mini`) を試してください。

### 4.4 Claude Code CLI を使う

ローカル PC にインストールされている `claude` バイナリ (Anthropic の Claude Code CLI) を介して呼び出します。Claude のサブスクリプション (定額契約) を持っている場合に有用で、追加の API 課金が発生しません。

1. ターミナルで `claude` コマンドが動く状態にしておく (未インストールならアプリ画面に表示されるインストール手順に従う)
2. LLM Wiki の **設定 → LLM モデル → Claude Code CLI** の行を ON
3. **CLI ステータス** が「検出済み」と表示されることを確認
4. API キーは不要

> ⚠️ Claude Code CLI モードはローカルサブスクリプションを利用します。ネットワーク接続と Claude へのログイン状態が必要です。

### 4.5 OpenAI 互換エンドポイントを使う

LM Studio、vLLM、LiteLLM、Azure OpenAI などの OpenAI 互換 API を持つサービスに接続できます。

1. **設定 → LLM モデル → Custom (OpenAI 互換)** を ON
2. **API モード** で「OpenAI 互換」または「Anthropic 互換」を選ぶ
3. **エンドポイント** に URL を入力 (例: `http://localhost:1234/v1`)
4. 必要なら API キーを入力 (ローカルモデルの場合は空欄でも可)
5. モデル名を入力

設定した内容は即座に保存されるので、別の章へ移動して大丈夫です。

## 5. プロジェクトを作成する

### 5.1 新規プロジェクトの作成手順

1. ウェルカム画面で **新規プロジェクト** をクリック
2. **プロジェクト名** を入力 (例: `my-research-wiki`)
3. **親ディレクトリ** を選択 (Wiki ディレクトリを置きたい場所の親フォルダ)
4. **テンプレート** を 5 種類から選ぶ
5. **作成** をクリック

> 💡 親ディレクトリは Obsidian の Vault (英: Vault、Obsidian がノートを管理する単位フォルダ) と兼ねるのがおすすめです。例えば `~/Documents/Obsidian/` を親ディレクトリにしておけば、後で同じディレクトリを Obsidian で開けます ([12 章参照](#12-obsidian-と併用する))。

### 5.2 5 つのテンプレートの選び方

シナリオが明確なら、それに合うテンプレートを選んでください。

- **リサーチ** 🔬 — 論文や記事を整理して、進化していく仮説 (ワーキング仮説) を追跡したい
- **読書** 📚 — 一冊の本について、登場人物・テーマ・章メモを構造化したい
- **自己成長** 🌱 — 目標・習慣・週次振り返り・ジャーナルを管理したい
- **ビジネス** 💼 — ミーティング、意思決定 (ADR 形式)、プロジェクト、ステークホルダーを文書化したい
- **汎用** 📄 — 用途を絞らず始めたい / 後で決めたい

### 5.3 作成後のディレクトリ構造

作成直後はだいたい次のような構造になっています (テンプレートにより一部違います)。

```
my-research-wiki/
├── purpose.md              # Wiki の目的・問い (人間が編集してもよい)
├── schema.md               # Wiki の構造ルール (基本は触らなくて OK)
├── raw/
│   ├── sources/            # ソースを置く場所 (PDF, Markdown 等)
│   └── assets/             # 画像など
├── wiki/
│   ├── index.md            # 全ページのカタログ (LLM が自動更新)
│   ├── log.md              # 操作ログ (LLM が自動追記)
│   └── overview.md         # Wiki 全体の俯瞰 (LLM が再生成)
├── .obsidian/              # Obsidian 用設定 (自動生成)
└── .llm-wiki/              # アプリ内部用 (チャット履歴、レビュー等)
```

> ⚠️ `.obsidian/` と `.llm-wiki/` は隠しディレクトリです。中身を直接編集する必要はありません。

## 6. ソースを追加する

「ソース」とは、Wiki のもとになる **生の資料** (PDF、Word、Markdown、画像、Web 記事など) のことです。

### 6.1 ファイル / フォルダをインポートする

1. サイドバーの **ソース** (本のアイコン) を開く
2. **インポート** ボタンをクリック
3. **ファイルをインポート** または **フォルダをインポート** を選ぶ
4. ファイル選択ダイアログで対象を選択

フォルダ単位でインポートすると、ディレクトリ構造を保ったまま再帰的に取り込まれ、フォルダ名 (例: `papers/energy/`) が LLM の分類ヒントとして使われます。

### 6.2 対応形式

| 形式 | 拡張子 |
|---|---|
| PDF | `.pdf` |
| Word | `.docx` |
| PowerPoint | `.pptx` |
| Excel / OpenDocument | `.xlsx`, `.xls`, `.ods` |
| テキスト・Markdown | `.txt`, `.md` |
| 画像 | `.png`, `.jpg`, `.gif`, `.webp`, `.svg` 等 |
| 動画・音声 | アプリ内蔵プレイヤーで再生可 |
| Web クリップ | Chrome 拡張からの Markdown |

### 6.3 取り込み中の挙動

ファイルを追加すると、画面下部の **アクティビティパネル** (取り込みキュー) に進捗が表示されます。

LLM が裏側で行うのは大まかに以下の 2 段階です。

1. **分析** — ソースを読んで、登場するエンティティ (英: entity、人物・組織・ツール等の固有名詞)、概念、既存 Wiki との関連、矛盾点を抽出
2. **生成** — 分析結果をもとに Wiki ページ (要約、エンティティページ、概念ページ) を作成・更新し、`index.md` と `log.md` を更新

複数ファイルを同時に投入しても、内部のキューが直列処理してくれるので待っているだけで済みます。失敗したタスクは最大 3 回まで自動リトライされます。

> 💡 同じファイルを再投入しても、内容が変わっていなければ SHA256 ハッシュ (英: hash、ファイル内容を一意に表す指紋のような短い文字列) で検出してスキップされます。トークン (= LLM の課金単位) を無駄にしません。

### 6.4 Chrome 拡張で Web 記事をクリップする

別途 Chrome 拡張をインストールすると、ワンクリックで開いている Web ページを Wiki に取り込めます。

1. `chrome://extensions` を開く
2. 右上の **デベロッパーモード** を ON
3. **パッケージ化されていない拡張機能を読み込む** をクリック
4. リポジトリの `extension/` ディレクトリを選択
5. 取り込みたい Web ページで拡張アイコンをクリック → プロジェクトを選んでクリップ

クリップは LLM Wiki アプリ側のローカル HTTP サーバー (ポート 19827) 経由で受信され、自動的に取り込まれます。

> ⚠️ クリップを受信するには LLM Wiki アプリが起動している必要があります。

## 7. wiki に質問する

ソースが取り込まれて Wiki ページが何ページかできたら、いよいよ質問できます。

### 7.1 チャット画面の使い方

1. サイドバーの **Wiki** タブで Wiki が開かれている状態にする
2. 中央のチャット欄にメッセージを入力
3. **Enter** で送信 (Shift+Enter で改行)

LLM は質問に応じて、関連する Wiki ページを自動的に拾い、引用つきで回答します。

### 7.2 良い質問のコツ

- **具体的に書く** — 「○○について教えて」よりも「○○と△△の違いを表でまとめて」など、形式や観点を指定すると精度が上がります
- **一度に詰め込みすぎない** — 大きな問いは小さな問いに分解しましょう
- **期待する形式を伝える** — 「箇条書きで」「比較表で」「3 行で」など

### 7.3 引用 (`[1]`, `[2]`) の見方

回答中に登場する `[1]` `[2]` のような番号は、参照した Wiki ページへのリンクです。回答下部の **ソース** セクションを展開すると、引用元のページタイトルとアイコンが一覧表示されます。クリックするとそのページがプレビューに表示されます。

### 7.4 良かった回答を「Wiki に保存」する

回答メッセージの下にある **wiki に保存** ボタンを押すと、その回答が `wiki/queries/` ディレクトリにアーカイブされ、自動的に Wiki に取り込まれます。これで、その質問の答えが知識ネットワークの一部として蓄積されます。

> 💡 **「探究」自体を Wiki に蓄積する** — これが LLM Wiki の核心です。チャット履歴に流すのではなく、価値のある回答は Wiki に保存して再利用できる資産にしましょう。

### 7.5 IME 利用時の注意

日本語入力で文字を変換中に Enter を押しても、メッセージは送信されず変換確定だけが行われます。変換確定後にもう一度 Enter を押すと送信されます (v0.4.0 で修正済み)。

> ⚠️ もし以前のバージョンから更新した直後にこの挙動が起きない場合は、アプリを完全終了 → 再起動してみてください。

## 8. 検索する

サイドバーの **検索** タブから、Wiki ページと元ソースを横断検索できます。

### 8.1 基本の使い方

1. **検索** タブを開く
2. キーワードを入力して **Enter**
3. ヒットしたページがリスト表示されるのでクリックしてプレビュー

### 8.2 日本語クエリの精度向上 — 形態素解析モード

検索エンジンは入力文をいったん単語に分割します。日本語は単語の区切りが明示されないため、**形態素解析** (英: morphological analyzer、文を意味のある単位に分割する処理) のエンジンを選べます。

| モード | 特徴 |
|---|---|
| **Auto (Intl.Segmenter)** | OS 内蔵の軽量分割エンジン。高速で追加バイナリ不要。既定。 |
| **Lindera (MeCab 互換 / 高精度)** | IPAdic 辞書同梱。活用形を吸収するため、「走った」で「走る」のページもヒット。アプリのバイナリサイズが約 50MB 増。 |

切替は **設定 → 情報 → 日本語トークナイザ** で行います。

> 💡 日常用途なら Auto で十分です。研究用途で精度を上げたいときに Lindera を試してみてください。初回切替時は辞書のロードで数秒かかります。

### 8.3 ベクター検索 (任意)

キーワード検索だけでは「同義語が違うので拾えない」ことがあります。**ベクター検索** (英: vector / semantic search、文の意味の近さで検索する仕組み) を有効にすると、表現が違っても意味が近いページを発見できます。

1. **設定 → 埋め込み** を開く
2. **ベクター検索を有効化** を ON
3. **エンドポイント** に OpenAI 互換の `/v1/embeddings` URL を入力 (LM Studio や Ollama Embedding モデルのローカルでも可)
4. **モデル** に埋め込みモデル名を入力
5. **全ページを再インデックス** をクリック

> ⚠️ 埋め込みモデルは LLM とは別のモデルを使います。ローカルなら `mxbai-embed-large` などが軽量です。

## 9. 知識グラフを眺める

サイドバーの **グラフ** タブを開くと、Wiki 全体のページを **ノード** (点) と **エッジ** (線) で可視化できます。

### 9.1 グラフの見方

- **ノードの色** — ページタイプ別 (entity, concept, source 等) または Louvain アルゴリズム (英: Louvain、グラフ理論で密につながった集団を発見する手法) による自動コミュニティ分け
- **ノードのサイズ** — リンク数が多いほど大きく
- **エッジの色 / 太さ** — 関連度が強いほど濃い緑、弱いほど薄い灰色
- **ホバー** — 隣接ノードがハイライトされ、関連度スコアが表示される

### 9.2 操作

- ノードをクリック → そのページがプレビューに表示される
- ズームイン / アウト / 画面フィット ボタンで全体俯瞰や拡大が可能
- 凡例 (右側) でタイプ別かコミュニティ別かを切り替え

### 9.3 グラフインサイト

グラフタブには **意外なつながり** と **知識のギャップ** という 2 種類の自動分析パネルがあります。

- **意外なつながり** — 異なるコミュニティをまたぐリンクや、ハブと周辺の組み合わせなど、目立つ関係を浮かび上がらせる
- **知識のギャップ** — 孤立ページ (ほぼ誰ともリンクしていない)、薄いコミュニティ (内部の相互参照が少ない)、ブリッジノード (3 つ以上のクラスタを橋渡しする) を提示

カードをクリックすると該当ノードがハイライトされ、**Deep Research ボタン** からそのギャップを埋める Web 調査を起動できます (次章参照)。

## 10. Deep Research (オプション)

LLM が知識のギャップを発見したときに、自動で Web を調べて結果を Wiki に統合する機能です。

### 10.1 事前準備 — Tavily API キー

Web 検索には [Tavily](https://tavily.com) という検索 API を使います。無料枠があります。

1. Tavily に登録して API キーを取得
2. LLM Wiki の **設定 → Web 検索 → Tavily API キー** に貼り付け

### 10.2 実行の流れ

1. サイドバーの **Deep Research** を開く (またはグラフインサイトの「Deep Research」ボタン)
2. リサーチトピックを入力 (またはグラフインサイト経由で自動生成された案が出る)
3. **編集可能な確認ダイアログ** でトピックと検索クエリを微調整
4. 開始すると LLM が Web を検索 → 内容を抽出 → Wiki に統合
5. 結果は自動的に取り込まれ、エンティティ・概念ページが追加・更新される

> 💡 同時実行は 3 タスクまで。`<think>` ブロック対応モデル (DeepSeek, QwQ など) を使うと思考過程も折りたたみ表示されます。

---

## 11. レビュー / Lint (オプション)

### 11.1 レビュー — 人間の判断が必要な項目を確認

LLM はインジェスト中に「これは人間に判断してもらいたい」と感じた事項を **レビュー項目** として残します。

サイドバーの **レビュー** タブを開くと、各項目に対して以下のような事前定義アクションが提示されます。

- **Create Page** — 新しい Wiki ページを作る
- **Deep Research** — Web で深掘りする
- **Skip** — 無視する

LLM がインジェスト時に最適化された Web 検索クエリも事前生成してくれているので、「Deep Research」を選ぶだけで調査が走ります。レビューはインジェストをブロックしないので、好きなタイミングで処理できます。

### 11.2 Lint — Wiki の健全性チェック

サイドバーの **Lint** タブで **Lint を実行** をクリックすると、Wiki 全体の整合性を機械的にチェックします。

検出される代表的な問題:

- **孤立ページ** — どこからもリンクされていないページ
- **リンク切れ** — `[[wikilink]]` の宛先ページが存在しない
- **発リンクなし** — 他ページへのリンクがないページ
- **矛盾** — ページ間の主張がぶつかっている (セマンティック Lint を有効にした場合、LLM が判断)
- **情報が古い可能性** — 新しいソースに置き換えられた可能性のある記述
- **ページ不足** — 重要な概念が言及されているのに専用ページがない

> 💡 月に 1 回や、ソースを 10 件以上追加した区切りで Lint を実行すると、Wiki が荒れにくくなります。

## 12. Obsidian と併用する

LLM Wiki の Wiki ディレクトリは、そのまま **Obsidian** (英: Obsidian、Markdown ベースの人気ノートアプリ) の Vault として開けます。

### 12.1 役割分担

| ツール | 担当 |
|---|---|
| LLM Wiki | Wiki の自動生成・更新、AI チャット、Deep Research、Lint |
| Obsidian | Wiki の閲覧、グラフビュー、手動編集、テンプレート、プラグイン拡張 |

### 12.2 並行起動の手順

1. LLM Wiki でプロジェクトを作成 (例: `~/Documents/Obsidian/my-wiki/`)
2. Obsidian を起動 → **Open folder as vault** で同じパスを開く
3. 両方を画面の左右に並べて使う

LLM Wiki がページを書き込むと、Obsidian 側で即座に反映されます。グラフビューやデータビュー (Dataview) などの Obsidian プラグインも利用できます。

> 💡 `.obsidian/` ディレクトリは LLM Wiki が初回作成時に推奨設定で自動生成します。Obsidian で Vault を初めて開くときの初期セットアップを省けます。

## 13. 困ったときの Q&A

### 13.1 インストールできない / 起動しない

- **macOS で「開発元が未確認」と言われる** — Finder で `.app` を右クリック → 開く を 1 度実施
- **Windows Defender が警告を出す** — 「詳細情報」 → 「実行」を選ぶ
- **Linux の AppImage が起動しない** — 実行権限を付与: `chmod +x LLM-Wiki-*.AppImage`

### 13.2 API エラーが出る

- **401 Unauthorized** — API キーが正しいか確認 (前後の空白も要チェック)
- **429 Too Many Requests** — レート制限に達した。少し待つか、別プロバイダに切替
- **ネットワークエラー** — オフライン時は Ollama や Claude Code CLI のローカル系を選ぶ

### 13.3 IME 入力の Enter で誤送信されてしまう

v0.4.0 で修正済みです。アップデート後も再現する場合はアプリを再起動してください。

### 13.4 ファイルパスに日本語が含まれる場合

問題なく動作します (内部で Unicode セーフな文字列処理が行われており、CJK ファイル名でクラッシュしない設計です)。

### 13.5 Lindera に切り替えたら検索が遅くなった

初回検索のみ辞書ロードで数秒かかります。2 回目以降は速くなります。常時遅い場合は Auto に戻してください。

### 13.6 バージョンアップの確認方法

**設定 → 情報 → 更新チェック → 今すぐ確認** をクリックします。「起動時に自動で更新を確認」を ON にしておくと最大 6 時間に 1 回チェックします。

## 14. もう一歩進んだ使い方

### 14.1 `purpose.md` と `schema.md` を編集する

- **`purpose.md`** — 「なぜこの Wiki を作るのか」を書くファイル。中心の問い、ワーキング仮説、スコープ、成功基準を埋めると、LLM のインジェスト・回答の質が大きく向上します
- **`schema.md`** — Wiki の構造ルール。新しいページタイプを追加したい場合や、命名規則を変えたい場合に編集します

LLM はインジェストやクエリのたびに両ファイルを読むので、変更は即座に反映されます。

### 14.2 AI 出力言語を強制する

英語ソースから日本語 Wiki を作りたい場合などは、**設定 → 出力設定 → AI 出力言語** で「日本語」を選んでください。Auto のままだとソース言語に追従するため、英語 Wiki が生成されてしまいます。

### 14.3 設定の永続先

各 OS でアプリの設定が保存される場所は次のとおりです。

- **macOS**: `~/Library/Application Support/llm-wiki/`
- **Windows**: `%APPDATA%\llm-wiki\`
- **Linux**: `~/.config/llm-wiki/`

API キー、選択中のプロバイダ、UI 言語、コンテキストウィンドウなどがここに保存されます。

### 14.4 Wiki ディレクトリを CLI で操作する場合の注意

Wiki ディレクトリは普通の Markdown ファイル群なので、Git で管理したり、外部ツールで grep したりできます。ただし以下に注意してください。

- LLM Wiki アプリ起動中に外部ツールで大量の変更を入れると、競合して取り込みキューが乱れることがあります。アプリを終了してから一括編集するのが安全です
- `.llm-wiki/` 内のファイルはアプリ専用です。手動編集しないでください

### 14.5 コンテキストウィンドウの調整

**設定 → LLM モデル → コンテキストウィンドウ** で、AI に一度に渡すトークン量を 4K 〜 1M の範囲で設定できます。

- 大きくする = より多くの Wiki ページを参照できる (品質向上、ただしコスト増)
- 小さくする = 安価・高速 (薄い回答になりがち)

割合は **Wiki ページ 60% / チャット履歴 20% / Index 5% / システム 15%** で配分されます。

## 付録 A: 用語集

| 用語 | 説明 |
|---|---|
| **wiki** | 相互リンクされた Markdown ページの集まり。LLM が自動で書き、人間は読む |
| **Vault** | Obsidian がノートを管理する単位フォルダ。LLM Wiki の Wiki ディレクトリは Obsidian Vault として直接開ける |
| **ingest (取り込み)** | ソースを読み込んで Wiki ページを生成・更新する操作 |
| **lint** | Wiki の整合性を機械的にチェックすること。プログラミングの lint と同じ語源 |
| **morphological analyzer (形態素解析)** | 日本語などを意味のある単語単位に分割する処理 |
| **RAG** | Retrieval-Augmented Generation。質問のたびに資料を検索して回答する仕組み |
| **LLM** | Large Language Model。ChatGPT などの大規模言語モデル |
| **entity / concept** | Wiki のページタイプ。entity は名前を持つ固有のもの (人物・組織)、concept は考え方や手法 |
| **frontmatter** | Markdown ファイル冒頭に書く YAML 形式のメタデータ |
| **wikilink** | `[[ページ名]]` 形式の Wiki 内リンク記法 |
| **Louvain** | グラフ理論のコミュニティ検出アルゴリズム。密につながったノード集団を自動で発見する |
| **embedding (埋め込み)** | 文章を数値ベクトルに変換し、意味の近さを計算可能にする手法 |
| **token (トークン)** | LLM がテキストを処理する最小単位。課金や履歴長の単位にもなる |
| **IME** | Input Method Editor。日本語などの入力変換システム |
